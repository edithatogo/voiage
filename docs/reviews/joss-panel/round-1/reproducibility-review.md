## Manuscript and repository blockers

### 1. R EVPI is not executable after installation

The reviewer reproduced the native R failure recorded in the research-software
report. The R tests substitute a stub rather than exercising the real library.
The native path, package metadata, licence statement, and installed-package
test require correction.

### 2. The release-bound SBOM claim is false

The public release has an sdist, three wheels, checksums, and provenance
attestations. The SBOM is a temporary Actions artefact and no tag-bound SBOM
workflow run was found for v1.0.0.

Required evidence after correction:

- generate an SBOM for the reviewed tag;
- retain or attach it durably;
- checksum it;
- connect it to the release in a machine-verifiable manifest.

### 3. The reviewed revision is ambiguous

The paper names v1.0.0, but `main` contains substantial manuscript,
health-example, and JOSS-package changes after that tag. A Code and data
availability section should identify the reviewed release, commit, archive,
documentation revision, fixed seed, script, generated outputs, and
synthetic-data status. The final software release should include the
paper-supporting corrections.

### 4. The impact wording is too strong

The fixed-seed example and same-author integration are developer-produced
applications or demonstrations, not independent research adoption.

### 5. Registry status and runtime completeness are conflated

`voiageR` is indexed by r-universe but does not provide a working,
self-contained Rust-backed EVPI installation. The paper should distinguish
registry indexing from functional completeness.

### 6. Cross-language consistency is not yet demonstrated

Line 102 should not claim consistent behaviour while the R route fails.
Restore a stronger claim only after genuine R, Julia, and Python conformance
tests pass.

### 7. The independent reviewer's second exercise is underspecified

The independent-validation protocol asks a participant to choose any relevant
example. It should instead provide a versioned command, expected output,
runtime estimate, and troubleshooting path.

### 8. Availability prose is under References

Move release and archive information to `# Code and data availability`.

Suggested text:

> This paper describes release `[reviewed version]` at commit `[SHA]`. The
> Python package can be installed with `python -m pip install
> voiage==[version]`. The synthetic health example uses seed `20260723` and is
> generated by `scripts/generate_paper_health_example.py`; its plotted and
> machine-readable outputs contain synthetic data only. Release files,
> checksums, attestations, and the release-bound SBOM are available from the
> GitHub release, and the source is preserved in Software Heritage as
> `swh:1:snp:767efde24c97d9f6d730764c1b3bc1a91ba20c32`.

Use this only after replacing the placeholders and producing the stated
release evidence.
