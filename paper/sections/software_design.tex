\section{Software design and assurance}

\subsection{Layered architecture}

The stable runtime is organized as a directed set of layers. The
\texttt{voiage-domain} crate validates primitive values, sample matrices,
strategy collections, trial designs, and decision problems.
\texttt{voiage-numerics} implements binding-independent kernels and depends on
the domain and diagnostic contracts. \texttt{voiage-serialization} defines
versioned result envelopes and canonical JSON behavior. Leaf adapters expose
these components through the private PyO3 extension and the C ABI\@; core crates
do not depend on Python, R, or Julia.

Python validation and orchestration remain necessary where callers provide
high-level model functions, labeled arrays, plots, or reports. This is a thin
binding boundary rather than a second numerical core. The native extension
reports its Rust version, target triple, source revision, lockfile digest,
build identifier, and per-operation invocation information. The C ABI uses a
versioned symbol namespace, explicit status values, caller-owned inputs, and
defined allocation and error-transport rules.

\subsection{Numerical scope}

The Rust core implements stable EVPI and ENBS kernels; regression-based EVPPI\@;
stochastic, efficient-linear, moment-based, and regression EVSI components;
CEAF and confidence intervals; strong and extended dominance; value of
heterogeneity; and structural EVPI/EVPPI aggregation. Python model evaluators
remain responsible for executing user-supplied models before structural
aggregation. Optional dependencies such as JAX, plotting libraries, and web
integrations are isolated from the base installation and do not redefine the
stable core contract.

Invalid inputs fail with structured categories such as invalid value or
dimension mismatch. Numerical tests cover finite inputs, degenerate and
single-strategy cases, translation and scaling properties, deterministic
seeding, and rank-deficient regression designs. The project does not convert
these implementation checks into a claim that a user's decision model is
valid.

\subsection{Assurance and release evidence}

Correctness is supported by language-neutral fixtures, unit and integration
tests, property-based and fuzz tests, metamorphic checks, mutation testing,
coverage gates, Rust diagnostics, ABI checks, benchmark regression budgets,
and clean-install tests. Normative fixture records contain inputs, expected
outputs, tolerances, and provenance. Compatibility tests execute the same
foundational contracts in the Rust support crate and the available language
adapters. Property tests check relationships that should hold across many
generated inputs, while fuzz tests exercise malformed and extreme cases.
Benchmark budgets guard against material regression but are not presented here
as cross-hardware speedup claims.

The release process produces platform wheels, a source archive, checksums, a
software bill of materials, provenance evidence, and a signed GitHub release.
Documentation is maintained in Astro/Starlight. The public v1.0.0 release
includes CPython ABI3 wheels for macOS arm64, Linux x86\_64, and Windows x86\_64,
plus a source distribution and SHA-256 checksums. Python packages are also
published to PyPI\@; registry review for other ecosystems is independent of
repository readiness.

The manuscript follows software-citation guidance~\cite{smith2016software}.
Its canonical source is authored LaTeX. The repository builds a deterministic
source-only archive, audits the review PDF, compiles against the supported
arXiv TeX Live generations, and independently converts the source to a
semantic web document. These checks establish repository readiness, not
submission or acceptance by arXiv or a journal.
